3.1 Core Concepts Deep Dive: Cohesion & Binding

This section explores the fundamental mechanics that enable Polymorphism and robust design.

1Ô∏è‚É£ Cohesion (Design Foundation)

What is Cohesion?
Cohesion measures how strongly related the responsibilities of a class or method are.

  • High cohesion = one class does one job well.
  • Low cohesion = one class does many unrelated things.

‚ùå Low Cohesion (Bad)

class UserService {
    void login() {}
    void generatePDF() {}
    void sendEmail() {}
    void connectToDatabase() {}
}
Problems Hard to maintain, changes ripple everywhere, and difficult testing.

‚úÖ High Cohesion (Good)

class AuthService {
    void login() {}
    void logout() {}
}

class EmailService {
    void sendEmail() {}
}
Benefits Easier maintenance, clear responsibility, reusable & testable.

2Ô∏è‚É£ Binding (Core Execution Concept)

Binding = connecting a method call to the method implementation.

obj.method();  // which method body?

1. Static Binding (Compile-Time / Early Binding)

  • Bound before execution, during compilation.
  • Happens with: static, final, private methods, and Method Overloading.
class A {
    static void show() { System.out.println("A"); }
}
A.show(); // Compiler knows exactly what to call

2. Dynamic Binding (Runtime / Late Binding)

  • Bound during execution.
  • Happens with: Method Overriding and Parent reference ‚Üí Child object.
class A {
    void show() { System.out.println("A"); }
}
class B extends A {
    void show() { System.out.println("B"); }
}

A obj = new B();
obj.show();  // B.show() -> JVM decides at runtime
Key Rule Methods are bound using the object type, not the reference type (except static/private/final).

3Ô∏è‚É£ Binding Consequence #1: Polymorphism

One interface, many implementations. Example: A obj = new B();

Compile-time vs Runtime Polymorphism

Type Mechanism Binding
Compile-time Overloading Static
Runtime Overriding Dynamic

JVM Insight (Intermediate+)

  • JVM uses a Virtual Method Table (V-Table).
  • Object holds method references.
  • Overridden method replaces parent entry.

4Ô∏è‚É£ Binding Consequence #2: Method Resolution

How Java resolves a method call:

  1. Compiler checks method existence using reference type.
  2. JVM checks overridden version using object type.
  3. Final method body executes.

5Ô∏è‚É£ Binding Consequence #3: Variable Resolution

Critical Variables are NOT polymorphic. 
class A { int x = 10; }
class B extends A { int x = 20; }

A obj = new B();
System.out.println(obj.x); // Output: 10

Why? Variables use reference type. No overriding. Compile-time resolution only.

6Ô∏è‚É£ Type Casting

Exists because Reference type and Object type are different.

1. Upcasting (Safe, implicit)

A obj = new B(); // Child ‚Üí Parent

2. Downcasting (Unsafe, explicit)

B obj = (B) new A(); // ‚ùå Runtime Error (ClassCastException)

// Safe Downcasting
if (obj instanceof B) {
    B b = (B) obj;
}
Note Casting does NOT change the object. It only tells the compiler to treat the reference as a specific type.

7Ô∏è‚É£ Master-Level Mental Model

  • Reference Type ‚Üí compiler rules
  • Object Type ‚Üí runtime behavior
  • Method Type ‚Üí binding decision
Interview One-liners
  • ‚ÄúJava achieves runtime polymorphism through dynamic binding.‚Äù
  • ‚ÄúStatic methods are resolved using reference type.‚Äù
  • ‚ÄúVariables do not participate in polymorphism.‚Äù
